# 文档质量审查清单

**用途：** 在 design 阶段结束时和 implementation 阶段内，作为 **Document Reviewer** 的逐项检查清单。每个不通过的项生成一条 finding。

---

## 第一部分：架构设计文档审查

> 对应文件：`docs/architecture_design.md` 或 `docs/architecture.md`

### D1: Mermaid 图表

- [ ] 文档是否包含至少一个 Mermaid 代码块？
- [ ] 是否有系统组件图（`graph TD` 或 `graph LR`）？
- [ ] 是否有核心业务时序图（`sequenceDiagram`）？
- [ ] 如有数据库设计，是否有 ER 图（`erDiagram`）？
- [ ] 图表是否清晰表达了组件间的调用关系？

**不合格判定：** 纯文字描述复杂架构关系 → `critical`（阻断）
**合格标准：** 至少包含 1 个组件图 + 1 个时序图

---

### D2: 技术选型理由

- [ ] 每项技术选型（语言、框架、数据库、中间件）是否有明确理由？
- [ ] 理由是否具体（如"团队有 3 年 Spring Boot 经验"），而非泛泛（如"流行"）？
- [ ] 是否说明了该技术的适用场景和局限性？
- [ ] 选型表格是否使用了标准格式（类别/技术/版本/理由）？

**不合格判定：** 只列技术名称无理由 → `critical`（阻断）

---

### D3: 架构决策记录 (ADR)

- [ ] 文档是否包含至少一条架构决策记录？
- [ ] ADR 是否遵循标准格式：背景 → 可选方案 → 决策 → 理由 → 后果？
- [ ] 每个重要决策（数据库选型、缓存策略、通信协议等）是否都有对应 ADR？
- [ ] ADR 是否编号（ADR-001、ADR-002...）便于引用？

**不合格判定：** 无任何 ADR 条目 → `critical`（阻断）

---

### D4: 模块依赖规则

- [ ] 是否定义了系统分层架构（如 controller → service → repository）？
- [ ] 是否明确了每层允许的依赖方向？
- [ ] 是否明确了禁止的反向依赖（如 domain 禁止依赖 infrastructure）？
- [ ] 依赖规则是否有图表辅助说明？

**不合格判定：** 未定义分层和依赖方向 → `patch`（可补充）

---

### D5: 非功能性需求

- [ ] 是否包含**安全性**设计（认证、鉴权、数据加密、SQL 注入防护）？
- [ ] 是否包含**性能**目标（响应时间、并发量、缓存策略）？
- [ ] 是否包含**可扩展性**设计（水平扩展、插件化、配置外部化）？
- [ ] 是否包含**容错性**设计（熔断降级、重试策略、幂等设计）？

**不合格判定：** 四项中缺少两项以上 → `patch`（可补充）

---

### D6: 数据库设计概要

- [ ] 是否列出了核心业务实体（如用户、订单、商品）？
- [ ] 核心表是否有字段定义（至少包含字段名、类型）？
- [ ] 是否标注了关键索引？
- [ ] 实体间关系是否有说明（一对多、多对多）？

**不合格判定：** 核心实体完全无字段定义 → `patch`（可补充）

---

## 第二部分：技术规格文档审查

> 对应文件：`docs/tech-spec-*.md` 或功能级技术文档

### T1: 接口 JSON 示例

- [ ] 每个 API 接口是否有完整的 URL + Method 定义？
- [ ] 是否有 Request Body 的 JSON 示例（含字段说明）？
- [ ] 是否有 Response 的 JSON 示例（含正常和异常响应）？
- [ ] 字段是否标注了类型（string/integer/decimal）和是否必填？
- [ ] 是否标注了字段约束（maxLength、min/max、格式要求）？

**不合格判定：** 无 JSON 代码块 → `critical`（阻断）

---

### T2: 数据库字段定义

- [ ] 新建表是否提供了 SQL DDL 或等效表格？
- [ ] 字段是否包含类型和长度（如 VARCHAR(32)、DECIMAL(10,2)）？
- [ ] 是否定义了主键、唯一约束和普通索引？
- [ ] 是否有默认值和 COMMENT 说明？
- [ ] 修改表是否明确了 ADD/MODIFY/DROP 操作？

**不合格判定：** 无字段类型定义 → `critical`（阻断）

---

### T3: 核心逻辑流程

- [ ] 复杂业务逻辑是否有流程图（Mermaid sequenceDiagram/flowchart）？
- [ ] 或者是否有伪代码描述关键步骤？
- [ ] 流程中的分支条件是否清晰（if/else 的判断依据）？
- [ ] 事务边界是否标注？
- [ ] 异步处理是否标注（MQ 发送、异步回调）？

**不合格判定：** 只写"实现XX逻辑"无任何细节 → `critical`（阻断）

---

### T4: 异常处理定义

- [ ] 是否列出了该功能所有可能的业务异常？
- [ ] 每个异常是否有错误码？
- [ ] 每个异常是否有触发条件说明？
- [ ] 是否区分了客户端错误（4xx）和服务端错误（5xx）？
- [ ] 是否有处理建议（提示用户/重试/告警）？

**不合格判定：** 未列出任何异常定义 → `patch`（可补充）

---

### T5: 依赖影响范围

- [ ] 是否描述了上游依赖（本功能调用的服务/接口）？
- [ ] 是否描述了下游影响（本功能产生的事件/变更影响谁）？
- [ ] 依赖的接口是否标注了方法签名或事件名称？

**不合格判定：** 完全未描述依赖关系 → `patch`（可补充）

---
