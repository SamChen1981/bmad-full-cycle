# 少样本示例：架构决策记录 (ADR)
# 用途: 作为 LLM 生成 ADR 的参考范本, 展示"合格的 ADR 长什么样"

---

## ADR-001: 数据库选型

- **背景：** 系统需要存储结构化订单数据（强一致性要求）和半结构化用户行为日志（灵活查询要求）。日均数据增量约 50 万条订单、2000 万条行为日志。
- **可选方案：**
  1. **MySQL 8.0** — 成熟稳定，团队 5 人全部熟悉，但 JSON 查询能力较弱。
  2. **PostgreSQL 15** — 原生 JSONB 支持复杂查询，ACID 完整，但团队仅 2 人有实战经验。
  3. **MySQL + MongoDB 混合** — 结构化用 MySQL，非结构化用 MongoDB，但增加运维复杂度和数据一致性风险。
- **决策：** 选用 **MySQL 8.0** 作为主数据库。
- **理由：**
  1. 订单等核心业务对一致性要求极高，MySQL 的事务支持成熟可靠。
  2. 团队全员熟悉 MySQL，学习成本为零，项目周期紧（3 个月）无法承担技术学习风险。
  3. 行为日志通过 JSON 字段 + 索引解决，MySQL 8.0 的 JSON 函数已能满足当前查询需求。
  4. 如果未来日志查询需求复杂化，可以引入 Elasticsearch 作为读侧补充，不影响主库架构。
- **后果：**
  - 行为日志的复杂聚合查询性能可能受限，需要提前规划 ES 引入时间点。
  - 放弃了 PostgreSQL 的高级 JSON 查询能力，但避免了团队能力不匹配的风险。

## ADR-002: 缓存策略

- **背景：** 商品详情页 QPS 预估 3000+，数据库直读无法承受。需要引入缓存层降低数据库压力，同时保证数据一致性（商品价格修改后不能展示旧价格超过 5 秒）。
- **可选方案：**
  1. **本地缓存 (Caffeine)** — 速度最快（微秒级），但多实例间不共享，一致性难保证。
  2. **分布式缓存 (Redis)** — 多实例共享，支持过期和主动失效，但增加网络开销（毫秒级）。
  3. **二级缓存 (Caffeine + Redis)** — 本地缓存做 L1（短 TTL），Redis 做 L2（长 TTL），兼顾速度和一致性。
- **决策：** 选用 **二级缓存方案（Caffeine L1 + Redis L2）**。
- **理由：**
  1. 商品详情页属于高频读场景，L1 本地缓存可以过滤 80% 以上请求，大幅降低 Redis 压力。
  2. L1 设置极短 TTL（3 秒），配合 Redis Pub/Sub 主动失效，可以满足"5 秒内一致"的业务要求。
  3. 商品修改是低频操作（每天几百次），Redis Pub/Sub 的广播开销可以忽略。
- **后果：**
  - 引入了缓存一致性的维护复杂度，需要在商品更新接口统一触发缓存失效。
  - 极端情况下（Redis Pub/Sub 消息丢失），可能出现短暂旧数据，但 L1 的 3 秒 TTL 保证了最长延迟不超过 3 秒。

## ADR-003: 服务间通信

- **背景：** 系统拆分为 3 个微服务（订单、库存、支付），需要选择服务间通信方式。核心场景包括：同步的库存扣减（下单时必须立即确认）和异步的支付通知（不阻塞主流程）。
- **可选方案：**
  1. **全部 HTTP/REST** — 简单统一，但异步场景需要轮询，效率低。
  2. **全部 gRPC** — 高性能，但团队无 Protobuf 经验，调试困难。
  3. **同步 HTTP + 异步 MQ** — 同步调用走 REST，异步事件走消息队列，各取所长。
- **决策：** 选用 **同步 HTTP/REST + 异步 RocketMQ** 的混合方案。
- **理由：**
  1. 库存扣减等同步场景，HTTP/REST 直观易调试，Spring Cloud 的 OpenFeign 可以零学习成本接入。
  2. 支付回调等异步场景，RocketMQ 提供可靠消息投递、延迟消息和事务消息，满足金融级可靠性。
  3. 团队有 2 人熟悉 RocketMQ，且公司基础设施已部署 RocketMQ 集群。
- **后果：**
  - 维护两套通信方式，需要在设计规范中明确"什么场景用同步/异步"的判断标准。
  - RocketMQ 的消费者需要做好幂等处理（参考 R9 防御性编程）。
